# Documentation Standard

## Purpose

Documentation is part of the product. It should be maintained continuously and kept close to the work it describes.

## Rules

- Document the why, what, and how for public-facing systems.
- Keep documentation in the source repository when possible.
- Prefer a single source of truth rather than duplicated descriptions.
- Use README files, standards, and templates to make the project self-explanatory.
- Capture significant decisions in ADRs.

## Review

- Verify that documentation exists for new features and APIs.
- Confirm docs are updated alongside behavior changes.
- Ensure the documentation is easy to locate using the repository naming conventions.
